# 知识库（Knowledge Base）

> Post-MVP 北极星设计。知识库是北极星的**第二根支柱**(第一根是"懂我"的记忆系统)。
> 依赖:`storage-architecture.md`(vault 布局、`messages`、`config`)、`scheduled-tasks.md`(乙类投喂)、`conversation-scopes.md`(反应/话题深聊)。

---

## 1. 为什么要有知识库

产品有两根咬合的支柱:

- **支柱 A · 懂我**:记忆系统(画像 / 工作集 / episode),理解"我是谁、在做什么"。
- **支柱 B · 知识库**:为成为 **T 型通才**服务——广度铺多领域 + 本领域深挖。

**咬合点**:知识库不是"执行平台",而是**第二个输入源 + 第二个反馈回路**。我对知识的反应(收藏、评价)本身就是关于"我"的高质量信号,反哺支柱 A。**只要我对知识有回应,Agent 就更懂我。**

知识库**由 Agent 拥有**(vault 是真相源),物理上是一个 Obsidian 仓库,我随时能打开读改。

---

## 2. 两类入库:主动沉淀 vs 被动投喂

知识库内部有两种生命周期相反的内容,**绝不能同等对待**,否则会被海量没人理的投喂内容淹没。

| | 甲类 · 主动沉淀 | 乙类 · 被动投喂 |
|---|---|---|
| 来源 | 我在别处看到好文章,主动让它收藏 | 定时 script 任务推送(AI 日报、知识卡片) |
| 方向 | pull(我筛过了) | push(我没筛,多数会划过) |
| 落地 | 直接进 `Garden`(kept) | 先进 `Inbox`(待反应) |
| 晋升 | 已是 kept,无需晋升 | 我的反应 = 晋升闸门 |

```
vault/
  Inbox/<任务名>/YYYY-MM/<slug>.md   # 乙类，status: inbox，永不自动删
  Garden/YYYY-MM/<slug>.md           # 甲类 + 被晋升的乙类，status: kept
```

- **Inbox 永不自动删**:按月子目录让它堆着也能翻;不做过期清理。
- **晋升 = 把文件从 Inbox 物理移动到 Garden**(移动文件不违反"不改正文")。

---

## 3. 甲类:主动收藏一篇文章

1. **触发 = 自然语言**(DM 里说"收藏这篇 / 存一下 `<url>`")。这不违反"Agent 不自作主张"——是我**显式开口**。当前**只支持 URL 文章**;零散文本片段以后有需要再说。
2. Agent 识别意图 → 调 `fetch_article(url)` 抓正文清洗成 markdown → `save_to_garden`。
3. **直接落 `Garden/YYYY-MM/<slug>.md`,`status: kept`**,跳过 Inbox(我已筛过)。
4. 记忆信号:我只甩 URL → 轻信号(顶多"对 X 领域有兴趣");我顺带说了为什么收藏 → 蒸馏一条 episode(关于我的判断/兴趣,**不碰画像**)。

---

## 4. 乙类:定时投喂 → Inbox

由 `script` 类定时任务产出,框架投递到 `Inbox/<任务名>/YYYY-MM/` 并发卡到通知群。脚本契约、投递机制见 `scheduled-tasks.md`。

---

## 5. 反应 = 晋升闸门 = 反馈回路触发点(命门)

核心论点:**我对卡片有反应 → 它更懂我**。所以反应绝不能只"搬文件",**必须同时往记忆里喂一条信号**。

反应发生在通知群里回复一张卡片,**回复的"方式"本身就是深度选择器**(完整机制见 `conversation-scopes.md`):

| 回复方式 | 晋升 | episode 蒸馏 | 深聊 |
|---|---|---|---|
| **普通回复** | 立刻 | **立刻**(一次性,无持续 scope) | 否,通知群仅捕获 |
| **话题回复**(飞书 thread) | 立刻 | **延到话题空闲关闭时**一次 | 是,thread 内深聊 |

- **晋升与蒸馏解耦**:晋升在反应手势那一刻立即发生;episode 蒸馏对话题回复延到 scope 收尾,避免"首次回复蒸一条、之后又蒸一条"重复。
- **普通回复的 episode 来源**:普通回复是一条离散反应消息,蒸馏出的 episode 记 `source_message_id` = 该回复消息,`source_scope_id` = 通知群,时间窗等于回复消息时间;通过该消息的 `reply_to` 可回到原知识卡消息,再由 `messages.knowledge_path` 找到知识文件。
- **正负边界(第一版约定)**:第一版默认普通回复都是正向意图(收藏/认可)→ 触发晋升;**不做负面语义识别**。但产品语义上"负面普通回复不应晋升到 Garden"——将来若支持负面回复,负面评价**只蒸馏 episode,不执行 Inbox→Garden 晋升**。此处留注释,避免把"所有普通回复都晋升"固化成错误语义,同时不给第一版增加实现分支。
- **红线**:反应蒸馏出的记忆信号**只写 episode / 工作集,不写身份画像**——和日记轮同一条红线。画像只在周合并、出现跨周稳定信号(如连续三周都对某新领域有反应)时才动。
- **只要回复评价这一种**反应方式(不做"收藏按钮")。

## 6. 检索:知识怎么被用回来

一个只进不出的知识库是写死的存档。两根支柱在"使用时"在这里咬合。

### 6.1 边界

- **记忆(episodes/profile/working)= "关于我"** → 小、常驻、**注入 systemPrompt 快照**。
- **知识(vault)= "关于世界、被我筛过的"** → 大,**不进快照,按需 ripgrep 拉**。

DM、主题群、话题都挂 `grep_vault` / `read_vault` 读类工具(日记群已有 `search_diary`)。

### 6.2 行为:主动连线,但克制

第二大脑的魔力在于**替我想起我自己都忘了存过的东西、并和我这个人连起来**。所以选**主动**:话题撞上 vault 可能有沉淀时,Agent 自动 grep,把"我存过的知识"和"关于我的记忆"简短连起来——而不是等我开口才查(那只是个带搜索框的文件夹)。

克制纪律靠 soul 提示词:**只在话题明显可能有沉淀时才 grep;连出来要综合、要短,绝不把搜到的原文倒给我**。

## 7. 知识地图(knowledge index):常驻快照的"第三块"

让 Agent 知道 vault 里**有什么**,它才知道**何时该 grep**——这块知识地图是检索精度的前提。

快照从此有**两块常驻摘要**:

| 常驻摘要 | 是什么 | 深挖入口 |
|---|---|---|
| 记忆摘要(profile+working+近期 episode) | "我是谁、在做什么" | `search_diary` |
| **知识地图(wiki-index)** | "我都收藏/知道些什么" | `grep_vault` |

定性:知识地图**不是记忆**(记忆=关于我),是"关于我收藏了什么世界知识"的第三类常驻组件。

### 7.1 生成方式:builtin 批处理 sub-agent

知识地图喂的是 system prompt、是系统核心行为,**不该是"用户脚本写错就污染快照"的外围件**。所以做成**第三个 builtin**(与周总结、记日记提醒并列;script 类因此维持"只发通知群"不放宽,见 `scheduled-tasks.md`)。

复用周总结那套隔离 sub-agent 形态:独立 scope + 干净 session + 聚焦 prompt + 跑完 reset。

- **输入廉价**:默认只把 vault 里各文件的 **frontmatter(标题/tags/领域/brief)** 喂给它,**不读全文**——否则知识库长到几百篇时它自己的 context 就爆了。配 `read_vault` 工具,想看某篇详情再选读。
- **有界 <3000 token**:地图必须是**分层压缩的"地图"而非"目录"**——顶层按领域聚类 + 粗略内容 + 大致数量("AI Agent 23 篇、分布式 8 篇…"),具体靠 grep 进去挖。这是 Hermes"有界、强制合并"纪律搬到知识地图。
- **目标驱动 prompt**:
  > 你的目标是——维持一张压缩、可导航的知识地图,如实反映知识库里已有的内容,让(未来的我和)Agent 随时知道"我沉淀了什么、该往哪挖"。按领域聚类、压缩、点出簇与连接,而不是罗列文件。可在末尾极简附上盲区/薄弱领域提示(可选,非必须)。
- **产出投递**:builtin 有自己的规则,把地图写到一个约定的快照槽位文件(如 `vault/.index.md` 或 ROOT 下约定路径),snapshot 构建时读它注入。

### 7.2 触发:按内容量,而非纯时间

bursty 输入下(可能一天放五六篇、也可能五六天放一篇)纯定时不灵活,纯阈值又会让地图在低产期永久搁浅。所以:

> **新增 ≥ N 篇内容 或 距上次刷新 > T 天且有新内容,谁先到算谁。**(N=5,T=3 起步,跑一阵再调。)

"自上次以来新增多少"**零额外状态**:index 文件自身的 mtime = 上次刷新时间,数一下 vault 里 mtime 比它新的文件数即可。daemon 用低频 tick(如每小时)评估此谓词,满足才真跑 sub-agent。

### 7.3 冻结与陈旧

快照在 session 开始时冻结(吃 prefix cache),地图是"上次脚本跑完那一版";session 中途新存的文章不在地图里——无妨,Agent 仍能 grep 到全量 vault,地图只是向导不是真相源。

## 8. frontmatter 约定

写进 vault 的**每个文件都必须有 frontmatter**(满足知识地图从廉价元数据生成的前提)。

```yaml
---
title: 文章标题
domain: AI Agent           # 领域，供地图聚类
tags: [retrieval, memory]
brief: 一句话摘要
status: inbox              # inbox | kept
source_url: https://...    # 甲类/抓取来源
saved_at: 2026-06-19T...
pushed_message_id: om_xxx  # 乙类：推送卡片的飞书 message_id（接反馈回路）
reacted_at: 2026-06-20T... # 我反应/晋升的时间
my_note: 我为什么收藏/我的看法
---
（正文，Agent 永不修改）
```

**乙类的 frontmatter 由框架从脚本返回的结构化字段渲染**(脚本只 return `{title, domain, tags, brief, body, source_url?}`,框架**用 YAML serializer**(`yaml`/`js-yaml`,非字符串拼接)渲染 frontmatter + 正文,标题/brief/tags 含冒号、换行、`---`、中文都能正确转义,合法性由 serializer 保证),见 `scheduled-tasks.md`。

## 9. 知识工具集(Phase: post-MVP)

| 工具 | 作用 | 类型 |
|---|---|---|
| `fetch_article(url)` | 抓正文清洗成 markdown | 网络 |
| `search_web(query)` | 联网搜索(知识讨论用) | 网络 |
| `grep_vault(query, scope?)` | ripgrep 读 vault | 读 |
| `read_vault(path)` | 读单个文件全文 | 读 |
| `save_to_garden(...)` | 建文件(甲类收藏) | 写(建) |
| `update_frontmatter(path, fields)` | 改反应/tags/评价 | 写(改 frontmatter) |
| `promote(path)` | 移动 Inbox→Garden | 写(移动) |
| `write_lark_doc(...)` | 把当前讨论写成飞书云文档,**仅显式要求时调用** | 网络 |

> **写类工具里没有"编辑正文"**——只有"建文件 / 改 frontmatter / 移动文件"。"Agent 永不改正文"这条原则在**工具层被物理锁死**,不靠 prompt 自觉。中文 grep 用 ripgrep(字符级子串,1-2 字短词也命中,比 trigram FTS 还省事)。

---

## 10. 非目标

- 不做"收藏按钮"等额外反应形态;反应只走"回复评价"。
- 不自动删除 Inbox(投喂内容永久留存,靠月份目录组织)。
- 不写心理状态定性 / 人格结论;反应只留基于文本的观察。
- 不让 Agent 修改文章正文。
- 不把知识塞进常驻快照(只放压缩地图);全文靠按需 grep。
